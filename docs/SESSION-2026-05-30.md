### 3. Flag emojis null pour toutes les équipes (`47bd350`)
**Cause** : `extractIsoFromLogoUrl` cherchait `/flags/xx.svg` mais API-Football retourne `/football/teams/22.png` pour les logos d'équipes nationales.

**Solution** : nouveau fichier `apps/web/lib/team-mapping/team-iso2.ts` avec mapping FIFA alpha-3 → ISO2 pour 60+ équipes.
```ts
export const TEAM_ID_TO_ISO2: Record<string, string> = {
  BRA: 'BR', FRA: 'FR', GER: 'DE', WAL: 'GB', ...
}
export function teamIdToFlagEmoji(teamId: string): string | null { ... }
```

Le normalizer utilise maintenant `teamIdToFlagEmoji(idA)` en priorité, avec fallback sur l'ancienne méthode URL.

**Bootstrap** : fallback calculé à la volée pour les équipes dont `flag_emoji` est null en DB :
```ts
flag_emoji: ct.teams.flag_emoji ?? teamIdToFlagEmoji(ct.teams.id)
```
→ 30/30 équipes avec flag emoji immédiatement, sans re-sync.

### 6. `sync-fixtures` : "r is not iterable" en prod (`566d36f`)
**Cause** : `redisSet` utilisait un mauvais format pour l'API REST Upstash.

```ts
// Avant — stockait l'objet entier comme valeur Redis :
await fetch(`${url}/set/${key}`, {
  body: JSON.stringify({ value, ex: ttl }),  // ❌ Upstash stockait {"value":"...","ex":3600}
})

// Après — format array de commande Redis correct :
await fetch(url, {
  body: JSON.stringify(['SET', key, value, 'EX', ttl]),  // ✅
})
```

Résultat : `fetchWithCache` récupérait `{"value":"[...]","ex":3600}` depuis le cache → `JSON.parse` → objet non-itérable → `for (const fixture of fixtures)` → `TypeError: r is not iterable` (minifié).

**Fix additionnel** : paramètre `isValid` sur `fetchWithCache` pour invalider les valeurs corrompues du cache existant :
```ts
return fetchWithCache(cacheKey, 3600, fetcher, Array.isArray)
```

### 7. `sync-fixtures` : FK violation `matches_nation_a_fkey` (`56a3e16`)
Après correction du Redis, le sync tournait mais échouait avec :
```
insert or update on table "matches" violates foreign key constraint "matches_nation_a_fkey"
```

**Investigation** : DEN et TUN étaient bien dans `teams` (vérification DB directe), mais le RPC échouait. Cause trouvée dans `db/migrations/005_centralized_engine.sql` :
```sql
-- Schéma original (ancienne architecture)
nation_a    TEXT    NOT NULL REFERENCES nations(id),   -- ← pointe sur "nations", pas "teams"
nation_b    TEXT    NOT NULL REFERENCES nations(id),
```

La table `nations` contient les équipes hardcodées WC2026. Denmark, Tunisia, et beaucoup d'équipes WC2022 n'y sont pas.

Migration 010 (`CREATE TABLE IF NOT EXISTS matches`) n'avait pas recréé la table ni modifié les FK existants.

**Tentative de fix** : `ALTER TABLE matches DROP ... ADD CONSTRAINT ... REFERENCES teams(id)` — échec car les anciennes lignes offline (WC2026 : AUT, etc.) ne sont pas dans `teams`.

**Fix final** : dropper les FK sans en remettre. L'intégrité est garantie applicativement (teams upsertés avant le RPC).
```sql
-- db/migrations/011_fix_matches_fk.sql
ALTER TABLE matches
  DROP CONSTRAINT IF EXISTS matches_nation_a_fkey,
  DROP CONSTRAINT IF EXISTS matches_nation_b_fkey;
```

### 9. Seed des FIFA rankings (`507ee5c`, `d351413`)

**Problème** : `strength: 75` et `initial_price: 100` pour toutes les équipes (valeurs par défaut).

**Cause du matching raté** : le script cherchait d'abord par `api_team_id` ; les équipes ont un `api_team_id` en DB, mais le CSV n'a pas ces IDs → aucun match → fallback sur défaut.

**Fixes** :
1. Fallback name matching même quand `api_team_id` est set :
```ts
const byId   = ct.teams.api_team_id ? rankings.find(r => r.apiTeamId === ct.teams.api_team_id) : undefined
const byName = rankings.find(r => r.teamName === ct.teams.name || ...)
const r = byId ?? byName  // ← fallback
```
2. CSV `scripts/data/fifa-rankings.csv` créé avec les rankings FIFA novembre 2022 + noms alternatifs (IR Iran, Korea Republic, USA/United States)
3. Ajout de vérification d'erreurs dans la boucle write
